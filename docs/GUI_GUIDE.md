# GUI

## Transcript

Streaming-ответы, tool cards с аргументами и результатами, summary-сообщения, статусные notice. Повторный префейс модели после выполнения инструментов подавляется с учётом нечёткого совпадения (fuzzy matching), чтобы избежать дублирования текста при минимальных расхождениях.

---

## CLI Output Widget

Для `cli_exec` вывод команды (stdout/stderr) отображается в реальном времени прямо в карточке инструмента, по мере поступления чанков. После завершения карточка автоматически сворачивается (с короткой задержкой для успешных команд). Команды с ненулевым exit code, использующие его как часть нормального протокола (`grep`, `rg`, `vulture`, `pytest`, `diff` и др.), не помечаются как ошибка — вывод возвращается с префиксом `Exit Code: N`.

## Composer

- Вставка файлов через `Add files…`, drag-and-drop или clipboard paste
- `@`-mention popup с файлами и директориями текущего workspace (обновляется динамически)
- Нормализация текста перед отправкой: `\r\n` → `\n`, удаление control-символов
- Лимит 10 000 символов на запрос с inline-предупреждением при усечении

## Plan Mode

Кнопка **Plan** в composer (справа от model selector) включает режим планирования для следующего запроса.

- Модель анализирует задачу, сначала изучает код через read-only инструменты и затем собирает компактный, decision-complete план
- Мутации и деструктивные инструменты заблокированы на этапе планирования
- `request_user_input` допускается только для одного реально блокирующего решения; approval этим инструментом не запрашивается
- После построения плана runtime показывает обязательную compact review-карточку `choice_type='plan_review'`:
  1. `implement` — реализовать
  2. `revise` — внести изменения в план
  3. `rebuild` — перестроить план
  4. `cancel` — отменить
- Review-карточка показывает plan card перед действиями и использует компактную сетку кнопок; рекомендованное действие визуально выделяется
- Внутри plan card план группируется в три слоя: обзор, шаги, служебные секции (`Проверка`, `Риски`, `Предположения`) только при наличии данных
- Если модель уже вывела markdown-план в transcript перед interrupt, UI переносит этот markdown внутрь review-карточки и убирает дубликат из текущего turn
- План сохраняется в state как `current_plan`, а `plan_status` становится `pending_approval`; события `plan_progress` также передают `current_plan` для side panel
- Выбранный пункт возвращается в graph через `Command(resume={"choice_type": "plan_review", "choice": <выбор>, "feedback": <выбор>})`
- Во время выполнения плана side panel показывает отдельную progress panel: header со статусом, compact plan card, активный шаг, завершённые шаги и meta-секции
- Панель выполнения показывается для активных состояний (`approved`, `executing`, `failed`, `replan_pending`, `rebuild_requested`, `blocked`) и не сбрасывается между follow-up run/retry, пока план не перешёл в terminal state
- При `completed`, `rejected` или `cancelled` progress panel скрывается; Inspector не открывается автоматически
- Кнопка автоматически сбрасывается после отправки запроса

## Горячие клавиши

| Клавиша | Действие |
|---|---|
| `Enter` | Отправить сообщение |
| `Shift+Enter` | Новая строка |
| `Ctrl+N` | Новый чат |
| `Ctrl+B` | Показать / скрыть боковую панель |
| `Ctrl+I` | Info popup |
| `↑` / `↓` в пустом composer | История отправленных сообщений |
